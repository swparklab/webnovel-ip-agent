# 운영실(Operations Studio) — 사용자 정의 · 갭 분석 · 추가 기능

> `secret/deep-research-report2.md`(SF 특화 웹소설 에이전트 전략 보고서)를 이 레포의 실제 코드로 옮긴 결과 문서.
> 보고서의 결론 — *"이 에이전트는 ‘좋은 SF를 생성하는 모델’이 아니라, 플랫폼별 취향·규칙·노출 구조를 학습해 SF를 **패키징하고 운영**하는 시스템이어야 한다"* — 을 구현했습니다.

---

## 1. 이 솔루션은 누가 쓰는가 (사용자 정의)

이 도구의 **사용자(=고객)**는 *작품을 한 번 쓰고 여러 플랫폼에 올려 운영해야 하는* SF/장르 웹소설 **작가·1인 스튜디오·IP 운영자**입니다. 그들이 모델링해야 하는 **독자 페르소나**는 보고서가 정의한 5종입니다.

| # | 독자 페르소나 | 주 플랫폼 | 반응하는 것 | 도구가 해줘야 할 일 |
|---|---|---|---|---|
| 1 | 인간중심 쾌감형 | r/HFY | 인간성·인류의 강점/결함이 드러나는 순간 | 하드설정보다 "인간이 무엇을 해내는가"를 먼저 말하는 블럽 |
| 2 | 성장 추적형 | Royal Road · ProgressionFantasy | 훈련·단계 상승·시스템·성장의 명시성 | SF라도 "성장이 어떻게 측정되는가"를 보여주는 태깅 |
| 3 | 아이디어 검증형 | r/scifiwriting | "재밌는가"보다 "말이 되는가" | 생성이 아니라 **설정 검증 코파일럿** |
| 4 | 모바일 binge/일일연재형 | Webnovel · Naver · Kakao | 챕터 수·안정 업로드·무료 회차·랭킹 | 작품성보다 **연재 습관·접근성·진열 카피** |
| 5 | 사회적 상호작용형 | Wattpad · 팬덤 | 코멘트·관계·작가-독자 친밀도 | 댓글을 관계망으로 보는 커뮤니티 모듈 |

핵심: 사용자는 **한 명의 작가**지만, 그가 상대하는 시장은 **서로 규칙이 다른 5~9개의 노출면**입니다. 그래서 도구의 본질은 "글을 더 잘 쓰게" 돕는 게 아니라 **"같은 글을 플랫폼마다 다르게 설명·태깅·연재·대응"** 하게 돕는 것입니다.

## 2. 지금 순(純) 기능이 그 니즈를 다 들고 있는가 — 갭 분석

레포의 기존 파이프라인은 **제작실(Production Studio)** — 6개 생성 에이전트(Foresight → 세계관 → Plot → Draft → Reader Sim → OSMU)입니다. 이는 보고서가 말한 **Narrative Intelligence(서사 지능)** 층에 해당합니다.

그런데 보고서의 두 번째 결론은 명확합니다.

> *"당신이 만들어야 할 것은 ‘SF 전용 작가봇’이 아니라 **SF 서사 분석기 + 플랫폼 번역기 + 커뮤니티 매니저 + 전략 리포터**다."*

즉 **Platform Intelligence(플랫폼 지능)** 층이 통째로 비어 있었습니다.

| 보고서가 요구한 기능 | 기존 레포 | 상태 |
|---|---|---|
| 서사 생성 (Narrative Intelligence) | 제작실 6 에이전트 | ✅ 보유 |
| 6층 자동 태깅 분류체계 | 없음 | ❌ 갭 |
| 플랫폼 적합도 스코어링 (RR Rising Stars/Main, 한국 장르표면 약세) | 없음 | ❌ 갭 |
| 플랫폼별 패키징·번역 (HFY/RR/Webnovel/Naver/Kakao 제목·블럽·태그) | 없음 | ❌ 갭 |
| 한국형 SF 오버레이 (SF→현판·재난·시스템 진열어 번역) | 없음 | ❌ 갭 |
| 커뮤니티 반응 분석 (6축 좋아요/싫어요 + 리뷰 신뢰도) | 없음 | ❌ 갭 |
| 전략 리포터 (성공식/실패식 채점 + 주간 액션) | 부분(Reader Sim은 작품 내부 채점만) | ❌ 갭 |

**결론: 기존 순 기능은 사용자 니즈의 절반(앞단=창작)만 들고 있었고, 보고서가 "진짜 차별점"이라 못 박은 뒷단(운영=플랫폼 번역·커뮤니티·전략)이 0%였습니다.** 그래서 그 뒷단을 새로 만들었습니다.

## 3. 무엇을 업데이트했는가 — 운영실(Operations Studio) 5 에이전트

플랫폼 지능 층을 **운영실**이라는 두 번째 스튜디오로 추가했습니다. 제작실과 같은 오케스트레이터·스트리밍·저장·폴백 인프라를 그대로 재사용합니다.

| 단계 | 에이전트 | 입력 | 산출물 (보고서 근거) |
|---|---|---|---|
| 1 | **자동 태깅기** (Tagger) | 작품 메타·시놉시스·샘플 | 6층 분류체계 태그 + 안전/리스크 플래그 |
| 2 | **플랫폼 적합도** (Platform-Fit) | 1번 | HFY/RR/Webnovel/Naver/Kakao 적합도 0~1, RS/Main 진입 가능성, 미스매치 경고 |
| 2 | **반응 분석기** (Reaction) | 붙여넣은 리뷰·댓글 | 6축(속도·문장·캐릭터·로맨스·설정정합성·태그약속) 점수 + 리뷰 신뢰도 가중 + 즉시/장기 수정 분리 *(태깅과 병렬)* |
| 3 | **플랫폼 번역기** (Packaging) | 1·2번 | 플랫폼별 제목·120/400자 블럽·태그8 + 필수/금지 단어 + 한국형 SF 오버레이 |
| 4 | **전략 리포터** (Strategy) | 1·2·3번 | 성공식/실패식 채점 + KPI(1→2화 유지·follow율) + 주간 액션·우선순위 |

설계 근거는 모두 보고서의 표·프롬프트·로드맵에서 직접 인코딩했습니다(`lib/platform-intel.js`의 `PLATFORMS`, `TAXONOMY`, `PERSONAS`, `KR_SF_OVERLAY`, `SUCCESS_FORMULA`/`FAILURE_FORMULA`).

### 앞단(UI)에서 추가된 것
- 상단 **제작실 ↔ 운영실 토글**. 운영실로 바꾸면 입력 패널이 *세계관·인물* 대신 *핵심 태그·타깃 플랫폼·샘플 챕터·붙여넣은 리뷰* 로 전환됩니다.
- 운영실 전용 탭: 태깅 / 플랫폼 적합도 / 반응 분석 / 패키징 / 전략 리포트 + **운영 가이드**(분류체계·플랫폼 규칙·페르소나).

### 뒷단(API)에서 추가된 것
- `POST /api/run` 에 `pipeline: "production" | "platform"` 파라미터. 운영실은 platform 파이프라인을 탑니다.
- `GET /api/platform-meta` — 분류체계·플랫폼·페르소나·성공/실패식을 프런트에 공급.
- 키가 없어도 도는 **운영실 결정론적 폴백**(`lib/platform-local.js`).

## 4. 보고서 로드맵 대비 현재 위치

보고서의 MVP 우선순위 — ①장면/챕터 라벨 스키마 ②플랫폼 규칙·장르 맵 ③챕터 반응↔메타 연결 — 중 **①②와 ③의 입력 경로**를 이번 업데이트가 구현합니다. 다음 단계(v1)는 반응 데이터를 *학습*해 페르소나 가중치를 갱신하는 피처 스토어이며, 그건 저장된 운영실 리포트가 쌓이면 붙일 수 있게 설계해 두었습니다.
